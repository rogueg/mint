This spec tests summarizing the hardest kind of change: a feature (fully custom page styling) that is inseparable from a security fix (markdown could previously execute JavaScript via passed-through Svelte syntax). The diff is also noisy — a 500-line example dashboard, lockfile churn, and snapshot updates dwarf the actual policy code in `cli/sanitization.ts`. A good summary leads with both the feature and the vulnerability, explains the layered enforcement model, and calls out the one deliberate security tradeoff.

# Evaluation guidance
We're evaluating the written summary only. The agent should not make any changes.

The audience is an expert teammate who hasn't seen this change and wants to understand it without reading the diff. Evaluate:
* **accuracy** — every claim should be checkable against the diff. This one is easy to get subtly wrong (e.g. claiming inline `style=""` attributes are supported — they're explicitly dropped).
* **altitude** — explains the *policy* (what's allowed, what's rejected, and why), not a function-by-function tour of the AST validator.
* **surfacing what matters** — the JS-execution hole and the remote-CSS tradeoff are the two things a reader most needs to know. Missing either is a significant failure.
* **noise filtering** — the example dashboard and docs are evidence of the feature, worth a sentence each; the lockfile and snapshot updates are not worth mentioning.

Key things a good summary covers:
* the feature: `<style>` blocks and layout HTML now survive markdown compilation, with `class`/`id`/`role`/`aria-*`/`data-*` attributes; demonstrated by the new `examples/flights/control_room.md` and documented in `docs/base.md`.
* the vulnerability: Svelte expressions in markdown (e.g. `{alert(1)}` in a page body or component prop) used to reach the compiler and execute. Dynamic markup is now rejected at compile time.
* the two-layer enforcement in the new `cli/sanitization.ts`: sanitize-html still decides which tags/attributes survive (now including `style` and the global attrs, with component attrs filtered and `{` escaped), and a Svelte AST pass (`validateSvelteMarkup`) rejects anything executable — expressions, blocks, `{@html}`, directives, spreads, `<svelte:*>`.
* the narrow whitelist of generated expressions: `GrapheneQuery` name/code static template literals and `ECharts config` JSON literals — and that these are generated by the compiler, not authored.
* the CSS policy: styles are extracted into `<svelte:head>`, sanitized with postcss (drops `expression()`, `javascript:`, `behavior`, `-moz-binding`, comments, angle brackets), and *deliberately* allow remote visual resources (`@import`, fonts, images). Inline `style=""` attributes are dropped.
* the tradeoff: remote CSS resources mean pages can trigger requests to arbitrary hosts. A good summary flags this as a conscious decision and notes why it's bounded (CSS values are static, so query data can't be interpolated into URLs).

Things to penalize:
* missing the security fix entirely, or describing this as "just" adding style support.
* walking through `validateSvelteNode`/`validateSvelteAttributes`/etc. one by one instead of stating the policy.
* spending more than a sentence or two on the example dashboard's contents.
* failing to mention that errors moved from runtime to compile time (it reshaped several tests and changes the authoring experience).

## Stronger sanitization
We weren't blocking all js, so something like `{alert(1)}` would run. `cli/sanitization.ts` now enforces the boundary in two steps: 
* `sanitize-html` keeps only allowed tags/attrs and escapes `{` in component attrs
* `validateSvelteMarkup` rejects executable Svelte syntax: expressions, blocks, `{@html}`, directives, spreads, and `<svelte:*>`. 

Invalid markup now fails at compile time with line/caret errors.

## Allow more HTML
Can now use more html tags along with `class`, `id` attrs. Inline `style=""` attrs are still removed.

## Style tags
Page `<style>` blocks are moved into `<svelte:head>` and sanitized with postcss. The CSS sanitizer removes execution and breakout paths (`expression()`, `javascript:`, `behavior`, `-moz-binding`, comments, angle brackets), but keeps remote visual resources like `@import`, fonts, and image URLs.

## Flight control example
The new `flights/control-room.md` example shows the kind of custom dashboard this enables.
